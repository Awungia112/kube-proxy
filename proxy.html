kube-proxy

kube-proxy is a critical component of Kubernetes networking. It enables communication between Pods and Services by managing network rules that route traffic to the appropriate backend Pods. t supports different modes of operation, with the most common being Iptables and IPVS

Module 1: The Foundational Problem and The Kubernetes Solution

1.1 The Ephemeral Nature of Pods

In Kubernetes, Pods are the smallest deployable units of computing. They are, by design, ephemeral or mortal. This means:

  • A Pod can be destroyed and recreated during deployments, scaling events, or node failures.
  • Each Pod gets a unique IP address that changes on recreation.

The Core Problem: How can an application like frontend reliably talk to a changing backend Pod IP?

1.2 The Solution: The Kubernetes Service Object

  • Service provides a stable ClusterIP that abstracts backend Pods.
  • Uses label selectors to group Pods.
  • Clients use DNS like backend-service.default.svc.cluster.local.

1.3 The Missing Link: From Virtual to Real

ClusterIP is virtual. The Linux kernel doesn’t know how to route it. So how does a request to 10.96.10.20 get to a real backend Pod like 192.168.1.5?

This is the fundamental role of kube-proxy.

Analogy: A Service is like a company’s main phone number. Pods are employees. kube-proxy is the phone exchange that forwards calls.

Module 2: The Architecture and Core Function of kube-proxy

2.1 Where and How It Runs

  • Runs on every node via a DaemonSet.
  • Each instance manages that node’s network translation rules.

2.2 How It Gets Information

kube-proxy is a network rule programmer. It watches:

  • Service objects – stable IPs and ports
  • EndpointSlice objects – list of live backend Pods

2.3 What It Does with the Information

Once it receives updated state, it programs the host’s networking stack (like iptables or IPVS) to handle routing and load balancing.

This ensures the kernel handles traffic forwarding efficiently without kube-proxy being in the data path.

Module 3: The kube-proxy Operational Modes (The Deep Dive)

3.1 userspace Mode (Legacy/Deprecated)

This was the original mode and is now deprecated. Summary:

  • Listens on high ports locally
  • Traffic routed to kube-proxy which then opens new connection to backend Pod

Flow: Client Pod → Kernel (DNAT) → kube-proxy process → Backend Pod

Why it’s bad: Very inefficient – packets cross userspace boundary twice.

3.2 iptables Mode (The Long-time Default)

Efficient mode using Linux kernel’s Netfilter framework:

  1. Packet hits PREROUTING or OUTPUT chain
  2. Jumps to KUBE-SERVICES chain
  3. Then to KUBE-SVC-XYZ – the service-specific chain
  4. Implements probabilistic load balancing with KUBE-SEP-* chains

All routing happens in-kernel, no user-space forwarding needed.
